# 09 I2C Peripheral Init MX_I2C1_Init

## Function

`static void MX_I2C1_Init(void)`

Ownership: [GENERATED] structure with [DRIVER] HAL init calls.

---

## Why this function matters for IMU

All IMU I2C callbacks depend on `hi2c1` being correctly initialized.

If this init is wrong:

1. callbacks may run,
2. but bus communication fails.

---

## What this function sets

1. peripheral instance: `I2C1`
2. timing register value
3. addressing mode: 7-bit
4. own addresses for slave mode (not used as master)
5. general-call mode off
6. no-stretch mode config
7. analog filter config
8. digital filter config

---

## Settings to understand first

### Addressing mode

`I2C_ADDRESSINGMODE_7BIT`

Why relevant:

1. BMI270 uses standard 7-bit I2C addressing.

### OwnAddress fields

Set to 0 in your case because MCU acts as bus master, not addressed slave.

### Timing value

Generated by CubeMX based on clock tree and target I2C speed.

At beginner stage:

1. trust generated value,
2. adjust through CubeMX if speed changes.

---

## HAL calls in this function

1. `HAL_I2C_Init(&hi2c1)`
2. `HAL_I2CEx_ConfigAnalogFilter(...)`
3. `HAL_I2CEx_ConfigDigitalFilter(...)`

You mostly need to know:

1. what each call configures,
2. what parameters mean,
3. that return code is checked.

No need to know internals of HAL register writes yet.

---

## Where pin-level setup happens

Pin mux/open-drain/pull-up details are configured in MSP init (`stm32l4xx_hal_msp.c`), not inside this function body.

Important association for your project:

1. PB6 -> I2C1_SCL
2. PB7 -> I2C1_SDA

---

## What you must master

1. This init must run before any IMU callback usage.
2. Addressing mode and bus timing are key concepts.

## What can be black box

1. Exact HAL register-level implementation.
2. Cube-generated boilerplate structure.
